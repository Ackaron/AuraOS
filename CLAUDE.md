# 🕹️ CLAUDE.md | AuraOS Control Center

## 🎯 Core Directives

- **Spec-First**: Никакого кода и никаких архитектурных решений без актуальной спецификации в `SPECIFICATION.md`.
- **Roadmap-Driven**: Любая работа должна соответствовать текущей фазе из `ROADMAP.md`.
- **Atomic Actions**: Одна задача из `ROADMAP.md` = одна итерация изменений.
- **GSD Workflow**: Всегда работаем через субагентов в порядке **Planner → Executor → Verifier**.
- **Local-First AI**: Приоритет только на локальные модели и локальные инструменты. Основной путь — Ollama / LM Studio.
- **UI/UX Fidelity**: Интерфейс строго по `.rules/ui-ux.md`. Стандарты **21st.dev**, **Magic UI** и **Fluid Zen** — обязательны.
- **No Hidden Magic**: Любое важное решение должно быть наблюдаемым, объяснимым и воспроизводимым.
- **File-Driven Architecture**: Роли, скиллы, команды и правила живут в файлах, а не в коде.
- **Graceful Degradation**: Если что-то ломается, система должна переходить в безопасный fallback-режим, а не падать целиком.
- **Greco Philosophy**: Код должен быть надёжным, классическим и без лишнего overengineering.

## 🤖 Reactive Agent System

AuraOS использует встроенный реактивный агентный цикл для исполнения задач.

### Базовый цикл

1. Получить запрос.
2. Определить цель и scope.
3. Выбрать роль: **Planner**, **Executor** или **Verifier**.
4. Сформировать следующую операцию.
5. Выполнить действие.
6. Прочитать observation.
7. Повторить цикл до завершения.
8. Завершить задачу через `finish`.

### Правила исполнения

- Любая сложная задача должна проходить через Planner перед кодом.
- Executor делает только реализацию, без расширения scope.
- Verifier отвечает только за проверку, тесты и соответствие спецификации.
- Никакая роль не должна подменять другую.
- Одна сессия = одна цель + один ограниченный набор действий.
- Максимум шагов в одной итерации должен быть ограничен и предсказуем.

### Инструменты агента

Агент может использовать только разрешённые проектные действия:
- чтение файлов;
- поиск по файлам;
- создание и обновление файлов;
- создание каталогов;
- удаление временных артефактов;
- завершение задачи отчётом.

### Правила путей

- Использовать только относительные пути.
- Никаких абсолютных путей.
- Никаких записей вне project root.
- Любая файловая операция должна быть безопасной и предсказуемой.

### Правило завершения

- Каждая реактивная задача обязана завершаться явным итоговым отчётом.
- Незавершённые циклы недопустимы.
- Если задача не может быть завершена, нужно зафиксировать блокировку и предложить следующий шаг.

## 🧭 Project Lifecycle

### `/plan`
- Используется для постановки задачи и обновления `ROADMAP.md`.
- Результат: понятный scope, роль, этап и ожидаемый outcome.

### `/execute`
- Используется для реализации конкретной атомарной задачи.
- Результат: изменения в коде, конфиге или документации.

### `/ui`
- Используется для работ по интерфейсу.
- Результат: обновление визуального слоя и UI-компонентов.

### `/think`
- Используется для проектирования логики оркестрации.
- Результат: схема маршрутизации, ролей, моделей и контекстов.

### `/review`
- Используется для проверки соответствия спецификации.
- Результат: список замечаний, подтверждение или отказ.

### `/save`
- Используется для сохранения состояния и сессий.
- Результат: лог сессии в `vault/sessions/`.

### Правила lifecycle

- Нельзя смешивать планирование и реализацию в одной неразделённой итерации.
- Нельзя переходить к коду без актуального плана.
- Нельзя считать задачу завершённой без проверки.
- Нельзя обновлять визуальную часть до фиксации архитектурной основы, если задача касается ядра.

## ⚙️ Environment & Setup

### Стек проекта

- **Frontend**: React + Vite + Tailwind CSS.
- **Backend**: Rust (Tauri v2).
- **AI Bridge**: Ollama API и LM Studio.
- **DB**: SQLite (`auraos.db`).
- **Orchestration**: Rust-first core + typed event bridge.
- **Design System**: Fluid Zen.

### Базовая среда

- Локальный LLM-контур должен работать через `http://localhost:11434` или совместимый локальный endpoint.
- Основной тон работы: лаконичный, технически точный, без лишней болтовни.
- Все изменения должны быть совместимы с локальной файловой структурой AuraOS.
- Любые новые ключевые переменные должны быть описаны в `.env.example`.

### Инициализация машины

- Если `.env` отсутствует, нужно запросить у пользователя необходимые ключи и создать файл.
- Секреты не должны попадать в репозиторий вне `.env`.
- `.env` должен быть добавлен в `.gitignore`.
- Все локальные настройки должны быть воспроизводимы на новой машине.

## 🔐 Security & Privacy

### Local-first правило

- Никаких облачных зависимостей в базовом сценарии.
- Все модели, данные и контекст должны оставаться локально.
- Любая внешняя интеграция должна быть явной и опциональной.

### Secrets management

- API-ключи и токены хранятся только в безопасном локальном хранилище или `.env`.
- Нельзя логировать реальные секреты.
- Нельзя писать секреты в рабочие документы, логи или историю сессий.

### File safety

- Любая операция с файлами должна быть ограничена project root.
- Перед удалением нужно убедиться, что файл действительно временный или не нужен.
- Нельзя использовать абсолютные пути.
- Нельзя выполнять действия вне разрешённого периметра.

### MCP safety

- Все локальные MCP-сервисы должны регистрироваться явно.
- Инструменты должны проходить валидацию перед использованием.
- Если MCP недоступен, система должна продолжать работу в упрощённом режиме.

## 🧩 Working Rules

### Приоритеты работы

1. Сначала понять задачу и её место в roadmap.
2. Затем проверить спецификацию и правила.
3. Потом определить роль: Planner, Executor или Verifier.
4. Затем выполнить минимально необходимое изменение.
5. После этого проверить результат.
6. Затем зафиксировать итог в состоянии проекта.

### Scope control

- Не расширять задачу без явного основания.
- Не добавлять лишние фичи “заодно”.
- Не переписывать большие блоки, если нужен точечный фикс.
- Не откладывать проверку на потом.

### Documentation discipline

- Если меняется архитектура — обновить `SPECIFICATION.md`.
- Если меняется план — обновить `ROADMAP.md`.
- Если меняется рабочий контекст — обновить `STATE.md`.
- Если меняются правила поведения — обновить `.rules/`.
- Если появляется новый operating pattern — зафиксировать его в `vault/index.md`.

## 🧠 Agent Behavior Guidelines

### Planner
- Формулирует цель.
- Разбивает её на минимальные шаги.
- Определяет риски, зависимости и критерии завершения.
- Не пишет код.
- Не делает лишних изменений.

### Executor
- Реализует только утверждённый шаг.
- Не расширяет scope.
- Не меняет архитектуру без разрешения.
- Сохраняет совместимость со спецификацией.
- Оставляет код читаемым и минимальным.

### Verifier
- Проверяет корректность реализации.
- Смотрит соответствие `SPECIFICATION.md` и `ROADMAP.md`.
- Проверяет тесты, UI, поведение и ограничения.
- Фиксирует замечания и риски.
- Не внедряет новые фичи.

## 🎸 The Greco Philosophy

AuraOS должен ощущаться как **Greco Les Paul 1975**:
- надёжный;
- классический;
- без лишнего перегруза;
- без избыточной сложности;
- с ясной архитектурой и стабильным звуком.

Инженерный смысл философии:
- лучше прочный минимализм, чем хрупкая “умная” сложность;
- лучше предсказуемость, чем эффектность;
- лучше совместимость и стабильность, чем архитектурная мода;
- лучше простая реализация, которая работает, чем идеальная концепция без поставки.
