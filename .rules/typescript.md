# 💻 TypeScript Standards

## 1. Type Safety
- **Strict Mode**: Всегда используй строгую типизацию в конфигурации проекта.
- **No `any`**: Использование `any` категорически запрещено. Если субагент `verifier` обнаружит `any`, выполнение задачи должно быть остановлено.
- **Validation**: Для валидации всех входящих данных (API, формы, Server Actions) обязательно используй библиотеку **Zod**.

## 2. Structure & Modularity
- **Interfaces over Types**: Для описания структур данных и объектов отдавай приоритет `interface`.
- **Single Responsibility**: Один файл должен содержать только один компонент или одну логическую сущность.
- **Typed Actions**: Все Server Actions должны иметь четко определенные интерфейсы для Payload и Response.

## 3. Documentation & Naming
- **Naming Conventions**: 
  - `PascalCase` используется для компонентов, классов, типов и интерфейсов.
  - `camelCase` используется для функций, переменных и методов.
- **JSDoc**: Каждая публичная функция или сложный метод должны иметь JSDoc-описание параметров и возвращаемого значения.

## 🎸 Engineering Excellence
- **Reliability**: Логика типизации должна быть надежной и чистой, как классический тон Greco Les Paul 1975, без лишних "шумов" и переусложнений.
- **No TODOs**: Запрещено оставлять пустые функции, заглушки типов или комментарии "реализовать позже".
- **Code Integrity**: Код должен быть полностью типизирован и готов к сборке (`npm run build`) без ошибок.
