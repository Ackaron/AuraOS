# 🧪 Engineering Testing Standards

## 1. The Testing Pyramid
- **Unit Tests (Vitest/Jest)**: Обязательны для сложной бизнес-логики, утилит и функций обработки данных.
- **E2E Tests (Playwright)**: Главный инструмент верификации. Любая новая фича или экран должны иметь базовый сценарий проверки работоспособности (Happy Path).
- **Component Testing**: Проверка UI-компонентов на корректность пропсов и состояний перед интеграцией.

## 2. Playwright & Visual Audit
- **Visual Regression**: При использовании шаблонов `uipro` агент обязан делать скриншоты через Playwright для сравнения с эталоном.
- **Headless Mode**: По умолчанию тесты запускаются в фоновом режиме, но при отладке сложных анимаций Framer Motion используется графический режим.

## 3. Validation Rules
- **Zod First**: Входные данные (API, формы) проверяются схемами Zod. Тесты обязаны проверять поведение системы при невалидных данных (Edge Cases).
- **Database Logic**: Проверка RLS политик Supabase через интеграционные тесты для гарантии изоляции данных пользователей.

## 4. Quality Gates
- **Zero Failures**: Выполнение задачи считается незавершенным, если хотя бы один тест упал (`npm run test` или `npx playwright test`).
- **No Mocking Overkill**: Старайся использовать реальные данные или моки максимально близкие к структуре из `schema.prisma`.

## 🎸 The Greco Standard
- **Clean Tone**: Тесты должны быть быстрыми и понятными. Если тест сложно прочитать — значит, код переусложнен ("шумит") и требует рефакторинга.
- **Reliability**: Твои тесты — это страховка. Они гарантируют, что проект останется стабильным даже после масштабного обновления библиотек.
