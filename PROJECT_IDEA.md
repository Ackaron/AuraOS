# 💡 PROJECT_IDEA: AuraOS (The Cognitive IDE)

## 1. Проблема (The Pain)
- **Суть боли**: Современные инструменты разработки (VS Code, Cursor) перегружены визуальным шумом и навязывают облачные подписки. Разработчик-аналитик тратит время на настройку контекста для каждой новой задачи, а ценные знания («скиллы») остаются запертыми внутри отдельных чатов или папок, не перетекая из проекта в проект.
- **Метрики**: 
    - **Context Switching**: Потеря до 40% продуктивности при переключении между задачами разного стека.
    - **Privacy Risk**: 100% корпоративного кода при использовании облачных LLM уходит на внешние сервера.
- **Текущие костыли**: Десятки вкладок в браузере, хаотичные заметки в Obsidian, ручное копирование промптов и «забытые» локальные модели, которые простаивают без дела.

## 2. Решение (The Concept)

**AuraOS** — это локальная экосистема для кодинга, аналитики и автоматизации, где LLM адаптируется под задачу, а знания и инструменты живут в виде переиспользуемого **Skill Core** и динамических **субагентов**.

### Основная ценность

AuraOS решает сразу три проблемы:

- Модель не теряет важный контекст между проектами.
- Знания не нужно копировать вручную из чатов и старых промптов.
- Под каждую задачу можно быстро собрать правильную связку: модель + субагент + скиллы + правила проекта.

Итог: пользователь работает не с «одним чатом», а с **живой локальной когнитивной системой**, которая растёт вместе с ним.

---

## 2.1 User Journey: Skill Core

### Initialization

Пользователь открывает AuraOS, и система автоматически индексирует локальную папку со скиллами: наработки по GuitarOS, бизнес-анализу ИНТЦ, шаблоны промптов, инструменты и другие переиспользуемые знания.

### Task Routing

При вводе команды вроде `/logic` или `/code` AuraOS выбирает подходящую модель через Ollama или LM Studio.

- Для логики может использоваться одна модель.
- Для кода — другая.
- Для анализа — третья.

Маршрутизация происходит автоматически, чтобы не заставлять пользователя вручную думать о том, какая модель лучше подходит под задачу.

### Contextual Execution

Агент выполняет задачу не на полном хаотичном контексте, а только на релевантных фрагментах из глобального Skill Core.

Это даёт три эффекта:

- Меньше токенов.
- Быстрее ответы.
- Меньше шума и противоречий.

## 2.2 Dynamic Federated Agents

AuraOS использует **Dynamic Federated Agents (DFA)** — файловую архитектуру, в которой роли не зашиты в код, а определяются содержимым папки `.claude/agents/`.

### Initialization

При запуске AuraOS:

- индексирует папку со скиллами;
- сканирует `.claude/agents/`;
- подхватывает доступные роли и правила;
- собирает актуальную карту возможностей системы.

### Dynamic Role Discovery

В боковой панели **Model Router** автоматически отображаются все роли, найденные в `.claude/agents/`.

Это означает, что:

- новые роли можно добавлять без изменения ядра;
- ненужные роли можно удалять простым удалением `.md` файла;
- система всегда отражает текущее состояние проекта.

### Task Execution

Пользователь выбирает роль вручную или система делает это автоматически.

Дальше агент работает в **layered context** — по принципу слоёв:

1. Базовая логика AuraOS.
2. Инструкции выбранного субагента из `.claude/agents/`.
3. Активные скиллы из Skill Core.
4. Частные правила проекта из `.rules/` и `CLAUDE.md`.

Такой подход позволяет каждому субагенту работать в своей зоне ответственности, но при этом использовать общий набор знаний и инструментов.

---

## 2.3 Логика субагентов

Субагенты — это отдельные специализированные роли внутри AuraOS, которые берут на себя конкретные типы задач.

### Как они работают

Каждый субагент может:

- иметь собственную цель и стиль работы;
- использовать свою модель;
- получать только нужный контекст;
- применять общие скиллы проекта;
- соблюдать правила конкретного workspace или проекта.

### Когда использовать субагента

Субагент нужен, когда задача:

- сложная;
- требует отдельного контекста;
- может быть разделена на независимые части;
- должна выполняться без перегрузки основного агента.

Примеры:

- код-ревью;
- бизнес-анализ;
- исследование;
- генерация документации;
- отладка;
- поиск ошибок в проектных правилах.

### Почему это важно

Субагенты позволяют:

- изолировать рабочий контекст;
- не засорять главный диалог;
- ускорять обработку задач;
- строить действительно модульную AI-систему.

---

## 2.4 Skill Core

**Skill Core** — это глобальная библиотека знаний, процедур, шаблонов, инструкций и инструментов, которые доступны всем агентам в системе.

### Что хранится в скиллах

Скилл может содержать:

- описание назначения;
- инструкции по работе;
- шаблоны промптов;
- примеры;
- правила выполнения;
- локальные инструменты;
- ссылки на MCP-серверы;
- метаданные для индексирования.

### Как используются скиллы

Когда пользователь запускает задачу, AuraOS:

- определяет, какие скиллы нужны;
- извлекает только релевантные части;
- добавляет их в контекст нужного агента;
- не загружает лишнее.

Это делает знания **постоянно доступными**, но не перегружает модель.

### Главное отличие

Скилл — это не разовый фрагмент в чате, а переиспользуемый модуль знания, который можно применять во всех проектах.

---

## 2.5 Skill Installer — GitHub-First

**Skill Installer** позволяет ставить скиллы и MCP-серверы одним действием — просто по ссылке на GitHub-репозиторий.

### User Journey

Пользователь открывает AuraOS или конкретный проект, затем:

1. Открывает панель **Skills**.
2. Нажимает **+ Add Skill**.
3. Либо вводит команду:
   `/skills install https://github.com/user/skill-name`
4. При необходимости указывает ветку или тег.
5. AuraOS автоматически устанавливает ресурс.

### Что делает AuraOS автоматически

После установки система:

- клонирует репозиторий через `git clone --depth 1`;
- размещает его в `~/.auraos/skills/<repo-name>` или в portable app-data;
- парсит `skill.json`, `mcp.json` и `README`;
- обновляет глобальный **Skill Core Index**;
- показывает тумблер **ON/OFF**.

### После включения тумблера

Скилл становится доступен:

- во всех проектах;
- всем субагентам, если он им нужен;
- в автоматическом и ручном режимах;
- без ручного копирования контекста.

---

## 2.6 Использование скиллов

Скиллы используются не как отдельные документы, а как часть контекстного слоя при выполнении задачи.

### Как это происходит

1. Пользователь задаёт задачу.
2. AuraOS понимает тип запроса.
3. Система выбирает нужную модель и нужного субагента.
4. Из Skill Core подтягиваются только релевантные навыки.
5. Агент собирает ответ или выполняет действие.

### Что это даёт

- Уменьшается расход токенов.
- Ускоряется работа.
- Уровень повторяемости и качества выше.
- Один раз настроенные знания работают везде.

### Пример

Если один раз подключить:

- UI/UX Pro Max;
- Playwright;
- бизнес-логику ИНТЦ;
- GuitarOS-философию;

то эти знания будут доступны всем агентам в любых проектах без повторной настройки.

---

## 2.7 MCP-серверы

AuraOS поддерживает не только обычные скиллы, но и полноценные **MCP-серверы**.

### Что это даёт

MCP-серверы позволяют подключать:

- внешние инструменты;
- локальные сервисы;
- структурированные источники данных;
- дополнительные возможности для агентов.

### Как подключаются

При установке из GitHub AuraOS:

- распознаёт `mcp.json` или другие метаданные;
- запускает MCP локально;
- регистрирует сервер в Model Router;
- делает его доступным агентам.

### Почему это важно

Это превращает AuraOS не просто в интерфейс для модели, а в локальную оркестрационную систему, где знания, инструменты и роли живут в одной архитектуре.

---

## 2.8 Layered Intelligence

AuraOS строит итоговый промпт по принципу слоёв.

### Слои промпта

1. Базовая логика AuraOS.
2. Инструкции выбранного субагента.
3. Активные скиллы.
4. Правила проекта.
5. Локальные инструменты и MCP-сервисы, если они нужны.

### Зачем это нужно

Такой подход даёт:

- предсказуемое поведение;
- меньший шум;
- лучшую управляемость;
- возможность быстро масштабировать систему.

Вместо жёсткой монолитной логики получается **живой стек интеллекта**, который легко расширяется.

---

## 2.9 Преимущества

AuraOS даёт следующие преимущества:

- Полная локальность и приватность: ничего не уходит в облако.
- Гибкость: роли определяются файлами, а не жёстким кодом.
- Масштабируемость: новые скиллы и субагенты подключаются без перестройки ядра.
- Унификация: один Skill Core работает для всех проектов.
- Удобство: установка через GitHub-link в один шаг.
- Долговечность знаний: всё важное сохраняется и переиспользуется.
- Адаптивность: разные задачи могут идти через разные модели и роли.

---

## 2.10 Ключевая формулировка

**AuraOS — это локальная экосистема, где субагенты отвечают за исполнение, скиллы — за знания, а Model Router собирает всё в единый layered context.**

Это делает систему похожей на живой когнитивный мозг, который растёт вместе с пользователем и не теряет накопленный опыт.

## 3. Анализ рынка (Context)
- **Конкуренты**: 
    - **Cursor**: Отличный UX, но зависимость от подписки и облака.
    - **VS Code + Continue**: Мощно, но слишком громоздко и сложно в настройке оркестрации.
- **Наше отличие**: Полная приватность (Local-First), легкость интерфейса в стиле "Fluid Zen" и уникальная концепция **Shared Skill Core**, которая не привязана к конкретному репозиторию.

## 4. Целевая аудитория (Audience)
- **Сегменты**: 
    - **Tech PMs & Analysts**: Кому важна структура, безопасность данных и автоматизация рутины.
    - **Power Users**: Обладатели топового железа (RTX 5080 / 9950x3d), желающие использовать его потенциал на 100%.
- **Jobs-to-be-Done**: «Когда я начинаю новый проект, я хочу, чтобы мой ИИ-ассистент уже знал мои стандарты качества, музыкальную философию и бизнес-логику ИНТЦ, не заставляя меня объяснять это заново».

## 5. Экономия и Развитие (Value)
- **Ценность**: Радикальное ускорение разработки собственных проектов (GuitarOS, ИНТЦ) и безопасная работа с данными стратегических партнеров.
- **MVP Scope**: 
    - Интерфейс на Tauri + React.
    - Интеграция с Ollama (выбор модели).
    - Базовый Skill Core (индексация папки с Markdown-файлами).
- **Future Scope (v2)**: 
    - Визуальный граф связей между агентами.
    - Интеграция с Obsidian для синхронизации «базы знаний».
    - Автоматическое дообучение (LoRA) на локальных данных.

## 6. Архитектура и Стек
- **Стек**: **Tauri (Rust) + React (Vite)**. Rust обеспечивает нативную скорость и прямой доступ к ресурсам ПК (9950x3d), React дает гибкость для создания премиального UI.
- **AI-слой (Local Model Router)**: Оркестрация через Rust-каналы. Автоматическое распределение нагрузки между VRAM (RTX 5080) и CPU.
- **Дизайн-код**: Эстетика "Fluid Zen", используем ui-ux-pro-max-skill вместе с компонентами **21st.dev**, плавные анимации, поддержка кастомных цветовых тем (от черного до бирюзового).
- **Схема взаимодействия**: 
  `[User UI] <-> [Tauri Rust Core] <-> [Local LLM Engines (Ollama/LM Studio)] <-> [Skill Core (Local Files)]`.
